# Data Model Suggestion 2: Event-Sourced / Audit-First (CQRS)

> Project: Emergency Management Platform · Created: 2026-05-22

## Philosophy

This model treats every state change as an immutable domain event stored in an append-only event store. The event store is the single source of truth; all queryable read models (projections) are derived from replaying events. This architecture is paired with CQRS (Command Query Responsibility Segregation): commands write events to the store, while queries read from purpose-built materialised views that can be rebuilt at any time from the event history.

Emergency management is inherently temporal: "what was the resource status at 14:30?", "what decisions were made in the first hour?", "reconstruct the incident timeline for the after-action review." Traditional CRUD databases answer these questions poorly because they overwrite state. An event-sourced model preserves the complete causal chain of every incident from activation to demobilisation, making after-action reviews trivial and regulatory audit responses immediate.

This pattern is used in production by financial trading systems, healthcare record systems, and military command-and-control platforms where the complete history of decisions and state changes is non-negotiable. The event store also serves as the foundation for AI-powered analytics: an LLM can process the event stream to auto-generate situation reports, identify anomalies, and produce after-action report drafts.

**Best for:** Agencies that require complete audit trails, temporal reconstruction of incident state at any point in time, and AI-powered analytics over incident history.

**Trade-offs:**
- Pro: Complete, immutable audit trail — every change to every entity is preserved forever
- Pro: Temporal queries are native ("what was the resource assignment at hour 3?")
- Pro: Read models can be optimised independently for different query patterns (dashboard, reporting, search)
- Pro: Event replay enables AI training on real incident patterns and automated AAR generation
- Pro: New read models can be added without changing the write model (e.g., add a graph view later)
- Con: Higher implementation complexity — developers must understand event sourcing, projections, and eventual consistency
- Con: Read models are eventually consistent (typically milliseconds, but not instant)
- Con: Event schema evolution requires careful versioning — old events must remain readable
- Con: Storage grows continuously (events are never deleted); requires archival strategy
- Con: Debugging requires understanding the event stream, not just the current state

## Standards Alignment

| Standard | How It's Used |
|----------|---------------|
| NIMS / ICS | ICS command actions, assignments, and form completions captured as typed domain events |
| CAP v1.2 (OASIS) | CAP alerts stored as `AlertOriginated` events; alert lifecycle (update, cancel) as subsequent events |
| ISO 22320:2018 | Incident command/control events preserve the decision chain required by the standard |
| NIST SP 800-53 AU | Event store inherently satisfies AU-2 (Audit Events), AU-3 (Content of Audit Records), AU-10 (Non-repudiation) |
| EDXL-RM | Resource requests and fulfillments stored as events, enabling reconstruction of the full request lifecycle |
| GeoJSON (RFC 7946) | Location data in events serialised as GeoJSON; spatial projections use PostGIS |
| EDXL-SitRep | Situation reports generated by projecting over event streams for a given operational period |
| FEMA NIMS Resource Typing | Resource type reference data maintained in a static lookup; resource lifecycle events reference type codes |

## Event Store Schema

```sql
-- ============================================================
-- EVENT STORE (Append-Only, Immutable)
-- ============================================================

CREATE TABLE event_store (
    id              BIGSERIAL PRIMARY KEY,                  -- monotonically increasing sequence
    event_id        UUID NOT NULL DEFAULT gen_random_uuid(),-- globally unique event ID
    stream_id       UUID NOT NULL,                         -- aggregate ID (incident, resource, etc.)
    stream_type     TEXT NOT NULL,                          -- 'Incident', 'Resource', 'Alert', 'Organisation', 'User'
    event_type      TEXT NOT NULL,                          -- domain event name (see Event Catalogue below)
    event_version   INTEGER NOT NULL DEFAULT 1,            -- schema version of this event type
    payload         JSONB NOT NULL,                         -- event-specific data
    metadata        JSONB NOT NULL DEFAULT '{}',            -- correlation_id, causation_id, user_agent, ip
    tenant_id       UUID NOT NULL,
    actor_id        UUID,                                   -- user who caused this event (null for system events)
    occurred_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Critical indexes for event store queries
CREATE INDEX idx_es_stream ON event_store(stream_id, id);               -- replay a single aggregate
CREATE INDEX idx_es_stream_type ON event_store(stream_type, id);        -- replay all events of a type
CREATE INDEX idx_es_tenant ON event_store(tenant_id, id);               -- tenant-scoped replay
CREATE INDEX idx_es_event_type ON event_store(event_type, occurred_at); -- event-type analytics
CREATE INDEX idx_es_occurred ON event_store(occurred_at);               -- time-range queries
CREATE INDEX idx_es_actor ON event_store(actor_id, occurred_at);        -- user activity audit

-- Optimistic concurrency control
CREATE UNIQUE INDEX idx_es_stream_version ON event_store(stream_id, id);

-- Partition by month for performance (events accumulate rapidly during major incidents)
-- In production, use declarative partitioning:
-- CREATE TABLE event_store (...) PARTITION BY RANGE (occurred_at);
```

The event catalogue defines the canonical event types. Each event type has a versioned JSON payload schema.

## Projection Tables (Read Models)

Read models are built by event processors that consume the event stream and maintain queryable tables. These tables can be dropped and rebuilt from the event store at any time.

```sql
-- ============================================================
-- PROJECTION: CURRENT INCIDENT STATE
-- ============================================================

CREATE TABLE proj_incident (
    id                  UUID PRIMARY KEY,                   -- matches stream_id
    tenant_id           UUID NOT NULL,
    incident_number     TEXT NOT NULL,
    name                TEXT NOT NULL,
    incident_type       TEXT NOT NULL,
    status              TEXT NOT NULL,
    severity            TEXT NOT NULL,
    description         TEXT,
    location            GEOMETRY(Point, 4326),
    affected_area       GEOMETRY(Polygon, 4326),
    jurisdiction        TEXT,
    start_time          TIMESTAMPTZ NOT NULL,
    end_time            TIMESTAMPTZ,
    last_event_id       BIGINT NOT NULL,                    -- last processed event
    last_updated        TIMESTAMPTZ NOT NULL,
    UNIQUE (tenant_id, incident_number)
);
CREATE INDEX idx_proj_incident_tenant ON proj_incident(tenant_id);
CREATE INDEX idx_proj_incident_status ON proj_incident(status);
CREATE INDEX idx_proj_incident_location ON proj_incident USING GIST(location);

-- ============================================================
-- PROJECTION: CURRENT RESOURCE STATE
-- ============================================================

CREATE TABLE proj_resource (
    id                  UUID PRIMARY KEY,
    tenant_id           UUID NOT NULL,
    organisation_id     UUID NOT NULL,

    current_incident_id UUID,
    current_location    GEOMETRY(Point, 4326),
    last_event_id       BIGINT NOT NULL,

);
CREATE INDEX idx_proj_resource_tenant ON proj_resource(tenant_id);
CREATE INDEX idx_proj_resource_status ON proj_resource(status);
CREATE INDEX idx_proj_resource_incident ON proj_resource(current_incident_id);
CREATE INDEX idx_proj_resource_location ON proj_resource USING GIST(current_location);
```

## Temporal Query Examples

### "What was the resource status at a specific time?"

```sql
-- Reconstruct resource state at 2026-05-22T14:30:00Z
SELECT payload
FROM event_store
WHERE stream_id = $resource_id
  AND stream_type = 'Resource'
  AND occurred_at <= '2026-05-22T14:30:00Z'
ORDER BY id DESC
LIMIT 1;

-- Or for full state reconstruction:
SELECT event_type, payload, occurred_at
FROM event_store
WHERE stream_id = $resource_id
  AND stream_type = 'Resource'
  AND occurred_at <= '2026-05-22T14:30:00Z'
ORDER BY id ASC;
```

### "Show the complete decision chain for this incident"

```sql
SELECT
    e.event_type,
    e.payload->>'summary' AS summary,
    u.display_name AS actor,
    e.occurred_at
FROM event_store e
LEFT JOIN ref_user u ON u.id = e.actor_id
WHERE e.stream_id = $incident_id
  AND e.stream_type = 'Incident'
ORDER BY e.id ASC;
```

### "Generate AAR timeline from events"

```sql
SELECT
    e.event_type,
    CASE
        WHEN e.event_type = 'IncidentActivated' THEN 'Incident activated: ' || (e.payload->>'name')
        WHEN e.event_type = 'ICSPositionAssigned' THEN (e.payload->>'user_name') || ' assigned as ' || (e.payload->>'position_title')
        WHEN e.event_type = 'ResourceDeployed' THEN (e.payload->>'resource_name') || ' deployed'
        WHEN e.event_type = 'AlertOriginated' THEN 'Public alert issued: ' || (e.payload->>'headline')
        ELSE e.event_type
    END AS timeline_entry,
    e.occurred_at
FROM event_store e
WHERE e.stream_id = $incident_id
  AND e.stream_type = 'Incident'
ORDER BY e.occurred_at ASC;
```

### "How many resources were deployed in the first hour?"

```sql
WITH incident_start AS (
    SELECT occurred_at
    FROM event_store
    WHERE stream_id = $incident_id AND event_type = 'IncidentActivated'
    LIMIT 1
)
SELECT COUNT(*)
FROM event_store e, incident_start s
WHERE e.event_type = 'ResourceDeployed'
  AND e.payload->>'incident_id' = $incident_id::text
  AND e.occurred_at BETWEEN s.occurred_at AND s.occurred_at + INTERVAL '1 hour';
```

## Event Processing Architecture

```
Commands (Write Side)                    Queries (Read Side)
========================                 ========================
  User Action                              Dashboard Query
       |                                        |
  Command Handler                          Read Model (Projection)
       |                                        ^
  Domain Logic                                  |
       |                                   Event Processor
  Event Store (append)  ──────────────>   (subscribes via LISTEN/NOTIFY
       |                                   or change data capture)
  LISTEN/NOTIFY ───────────────────────>
```

### Event Processor (Pseudo-code)

```python
# Event processor listens for new events and updates projections
async def process_event(event):
    match event.event_type:
        case "IncidentActivated":
            await upsert_proj_incident(event)
            await append_proj_timeline(event, "Incident activated")
        case "ResourceDeployed":
            await update_proj_resource_status(event, "assigned")
            await append_proj_timeline(event, f"Resource deployed")
        case "AlertOriginated":
            await upsert_proj_cap_alert(event)
            await append_proj_timeline(event, "Alert issued")
        # ... handlers for all event types
```

## Key Design Decisions

1. **Single event store table** rather than per-aggregate tables. This simplifies infrastructure (one table to partition, back up, and replicate) and enables cross-aggregate queries (e.g., "all events in the last hour across all incidents"). Partitioning by `occurred_at` month keeps individual partitions manageable.

2. **JSONB payload** rather than typed columns per event. This allows the event schema to evolve without DDL migrations — new event types are simply new JSON shapes. The `event_version` column enables payload schema evolution with backwards-compatible deserialisers.

3. **Projections are disposable** — they can be dropped and rebuilt from the event store at any time. This is critical for emergency management where a new reporting requirement (e.g., FEMA asks for a new cost tracking view) can be met by adding a new projection without touching the write model.

4. **Reference data lives outside the event store** in conventional relational tables. Users, organisations, resource types, and ICS positions are not event-sourced because they change infrequently and are needed for JOIN queries against projections.

5. **Optimistic concurrency** via the monotonically increasing `id` column. When two users try to update the same incident simultaneously, the command handler checks the expected stream version against the actual latest event, rejecting conflicting writes.

6. **PostgreSQL LISTEN/NOTIFY** as the primary event distribution mechanism for projections. For larger deployments, this can be replaced with a dedicated event bus (Apache Kafka, NATS JetStream) without changing the event store schema.

7. **Snapshots** for long-lived aggregates. An incident that runs for weeks may accumulate thousands of events. Periodic snapshots capture the aggregate state at a point in time, allowing replay to start from the snapshot rather than the beginning.

8. **AI-native advantage**: the event store is the ideal input for LLM-based incident analysis. An AI agent can consume the event stream for an incident and produce structured outputs (AAR drafts, timeline summaries, anomaly detection) because every decision, action, and state change is preserved as a discrete, typed event with timestamps and actor attribution.
